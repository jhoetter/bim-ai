  ### WP 2 — FAM-01: Nested families (load-bearing slice)

  **Tracker entry:** `spec/workpackage-master-tracker.md` → "Strategic Primitive 4" → FAM-01 + the **FAM-01 detail** block.

  **Reality check.** The full FAM-01 spec is L (2-3 weeks of engineering). You will **not** finish all of it in one shift. Your job is to ship the **load-bearing slice** that lets KRN-09
   (WP 3 below) consume nested families. Concretely:

  **MUST ship (load-bearing — mark `done`):**
  1. **Data model.** Add `family_instance_ref` family-geometry node:
     ```ts
     {
       kind: 'family_instance_ref';
       familyId: string;
       positionMm: { xMm: number; yMm: number; zMm: number };
       rotationDeg: number;
       parameterBindings: Record<string, ParameterBinding>;
       visibilityBinding?: { paramName: string; whenTrue: boolean };
     }
     type ParameterBinding =
       | { kind: 'literal'; value: number | string | boolean }
       | { kind: 'host_param'; paramName: string }
       | { kind: 'formula'; expression: string };
     ```
     In `packages/web/src/families/types.ts` (or wherever family geometry types live — grep for the existing union). Mirror in Python if there's an equivalent.

  2. **Resolver.** A function `resolveNestedFamilyInstance(instanceRef, hostParams, familyCatalog)` that:
     - Looks up the nested family by `familyId`
     - Computes effective parameter values for the nested family by applying `parameterBindings` (literal / host_param resolution / formula via FAM-04's `evaluateFormula`)
     - Recursively resolves the nested family's geometry with those parameters
     - Returns the geometry tree (BufferGeometry or similar — match what other family geometry nodes return)

  3. **Cycle detection.** At family-save time and at resolution time, detect cycles in the nested-family graph (BFS up to a depth of 10). Reject saves that would introduce a cycle; throw
   a clear error at resolution time (defensive).

  4. **Engine + family-catalog hookup.** When the family resolver encounters a `family_instance_ref` node, it walks into the nested family's geometry. Wire this so existing
  geometry-rendering callers (door, window, etc.) continue to work transparently.

  5. **Tests:**
     - vitest: build a 2-level nested family (door containing a swing arc), assert the swing arc is rendered with the correctly-bound radius
     - vitest: build a cyclic nested-family graph, assert resolver throws / rejects
     - vitest: `parameterBindings` propagation (literal / host_param / formula via FAM-04)

  **MAY defer (mark these in the tracker note as "deferred to follow-up"):**

  - **Family editor UI for placing nested instances.** The "Loaded Families" sidebar + drag-drop from FL-06 into the editing canvas — defer. KRN-09 doesn't need a UI; it consumes nested
  families programmatically through the engine.
  - **Per-instance parameter binding editor in the inspector.** Defer. Bindings can be authored in JSON for now.
  - **Visibility-binding integration with FAM-03.** FAM-03 isn't shipped yet anyway; the data model field is in place for when it lands.
  - **Thumbnail generation for nested family compositions** in the FL-06 panel — defer.

  After this WP, mark FAM-01 in the tracker as `partial` with a note: `partial in <hash> — data model + resolver + cycle detection shipped; family-editor UI for placing nested instances
  + per-instance parameter editor deferred to follow-up`.

  **Acceptance.** A test fixture authoring a "DoorWithSwing" family that contains a nested 2D-swing-arc family resolves correctly: changing the door's `Rough Width` updates the swing
  arc's radius via the formula binding `Rough Width - 2 * Frame Width`. The nested family's geometry is included in the host family's resolved geometry tree.

  **Files:** `packages/web/src/families/types.ts`, `packages/web/src/families/familyResolver.ts` (likely new), `packages/web/src/lib/expressionEvaluator.ts` (FAM-04, read-only), tests in
