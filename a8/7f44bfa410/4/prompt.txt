## Design Principle: Native Integration with Existing Concepts

The Energy Lens is **not a bolt-on**. It must extend existing platform concepts naturally so that an Architecture-Lens user and an Energy-Lens user are working on the **same model, same elements, same wall types, same rooms** — but seeing different overlays, properties, schedules, and tools.

### How the Energy Lens extends existing concepts

| Existing Concept | Energy Lens Extension |
|---|---|
| **Wall types with layers** | Layers gain thermal properties (λ, ρ, c, μ). U-value computed automatically per wall type. No duplicate wall modeling — the architect's wall is the energy consultant's wall. |
| **Building elements** | Each element gains optional `thermalClassification` (e.g., "exterior wall against outside air", "floor against ground") and `thermalBridgeMarkers`. Architecture Lens hides these properties; Energy Lens surfaces them. |
| **Rooms/zones** | Rooms gain `heatingStatus` (heated / low-heated / unheated) and `usageProfile` (residential / office / school / etc. per DIN V 18599). Architecture Lens shows room name and area; Energy Lens additionally shows heating status, setpoint, and air change rate. |
| **Schedules** | New schedule types: "Envelope Surfaces", "Thermal Materials", "Building Services". They behave like existing Door/Room/Window schedules — same UI, same filtering, same export — but show energy-specific columns. |
| **Sheets** | Energy-relevant viewports (envelope visualization, U-value heatmap, shading analysis) can be placed on sheets exactly like architectural plan viewports. A consultant can produce a "Bestandsaufnahme" sheet with envelope summary the same way an architect produces a "General Arrangement" sheet. |
| **Sections** | Sections work unchanged but Energy Lens overlays layer thicknesses, λ-values, and U-values on the cut wall — turning a section into a thermal section. |
| **3D Views** | Existing saved views (axonometric, elevation) remain valid in Energy Lens. Additional view modes: "Color by U-value", "Color by thermal classification", "Highlight thermal envelope", "Shading heatmap over year". These compose with existing Wire/Shaded/Realistic graphics modes. |
| **Sun Studies** | Already present — extend with annual shading factor per window, used for solar gain assessment. |
| **Discipline Lens (today: visual filter)** | Becomes property-set-aware. Selecting "Energy" lens changes which properties appear in inspectors, which schedules open by default, which toolbar tools are prominent, and which view modes are exposed. Architecture and Energy lens never fight over the same screen real estate. |
| **Cloud collaboration** | Architect and energy consultant can work on the same model simultaneously, each in their own lens. Architect moves a wall — envelope area updates in the consultant's session in real time. Versioning allows "as-is" vs "renovation scenario A/B/C" as branches. |
| **API / third-party apps** | All energy data is queryable via API. Third-party developers can build apps (e.g., quick iSFP summary landing page, customer-facing 3D walkthrough with energy info, BAFA application helper) on top of the same model. |

### What this rules out by design

- No parallel "energy model" that the consultant has to keep in sync with the architectural model
- No separate material database for energy and non-energy materials — one material, optional thermal properties
- No proprietary file format for energy data — everything stays in the project, lens-filtered
- No special "energy mode" that disables architectural editing — both lenses are live simultaneously

Seed the platform with a standard library based on DIN 4108-4 (~300 common construction materials: mineral wool WLG 035, EPS WLG 032, sand-lime brick, reinforced concrete, gypsum board, OSB, etc.). Users can extend with custom materials.

A material without thermal properties remains valid for non-energy use — Architecture Lens never sees these fields, Energy Lens surfaces them when needed.
