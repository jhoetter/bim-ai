## WP entry (verbatim from tracker)

```
#### DSC-V3-01 — Discipline tags on elements

_Source: D4. Severity High. Foundation primitive — without per-element tags the lens has
nothing to filter on._

**Status.** `now`.
**Scope.** Every kernel element gains an optional `discipline?: 'arch' | 'struct' | 'mep'`
field. Default at create time is `'arch'` (the primary lens). Structural kinds (column, beam,
brace, foundation) default `'struct'`; MEP kinds (duct, pipe, fixture) default `'mep'`.
Architects can re-tag (a load-bearing wall → `'struct'`). Tags are commandable + undoable;
participate in the activity stream.
**Data model.**
  discipline?: 'arch' | 'struct' | 'mep';   // default 'arch' if absent

  DEFAULT_DISCIPLINE_BY_KIND: Record<ElementKind, 'arch' | 'struct' | 'mep'> = {
    wall: 'arch', door: 'arch', window: 'arch',
    column: 'struct', beam: 'struct', brace: 'struct', foundation: 'struct',
    duct: 'mep', pipe: 'mep', outlet: 'mep',
    // ... full table mirrors v2 element kinds
  };
**Engine.** `SetElementDiscipline` command — sets `discipline` on N elements; undo +
activity. New elements pull default at create. Validation: enum or null (treated as `'arch'`).
Schedule queries gain `where_discipline`. Phasing + design options stay orthogonal — an
element can be `phase_demolished + discipline: 'struct'` simultaneously.
**UI.** Inspector "Discipline" dropdown for any selected element (three options + a "default
for this kind" radio). Project Browser categories group gets discipline subgroups for
browsability.
**Acceptance.** Author a column → discipline auto-resolves `'struct'`. Re-tag a wall to
`'struct'` → activity records it. Lens "Show: Structure" includes the column and the
re-tagged wall.
**Effort.** M — 1.5 weeks.
**Cross-theme references.** T1 (kernel kinds get discipline as their default); T9 (agent's
tokenisation includes discipline as a token slot); T7 (toposolid is `arch`, not a fourth
discipline).
```

## Implementation rules (verbatim from §A)

- **Honour API-V3-01.** Every kernel command you add (`SetElementDiscipline`) must have
  CLI + REST/RPC + JSON schema + assumption-log support. Add tests at the agent-callable
  layer (`tests/cli/`, `tests/api/`).
- **No scope creep.** Do not silently add features not in this WP entry. DSC-V3-02,
  LNS-V3-01, and LNS-V3-02 are separate WPs — do not implement them here.
- **Design-pillar adherence.** This WP directly serves D4 (color is information) and D8
  (context, not configuration). The Inspector dropdown is the correct surface (D1 direct
  manipulation, D8 context-driven). Do not introduce a dialog or a global matrix (A8
  antidote).
- **No hex literals in `packages/web/src/**/*.tsx`.** All discipline tints must consume
  `var(--disc-arch)`, `var(--disc-struct)`, `var(--disc-mep)` and their `-soft` variants.
  The ESLint rule (R-G §2.5 phase 3) will fail the build otherwise. The Inspector dropdown
  may show the tint chip using the CSS variable only.
- **Phasing + design options are orthogonal.** An element holds `discipline` independently
  of `phase` and `designOption`. Do not couple them.
- **Activity stream participation.** `SetElementDiscipline` must emit an activity-stream
  entry (same pattern as other commandable properties). Undo must reverse it.
