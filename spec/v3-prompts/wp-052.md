## Verbatim EDT-V3-04 tracker entry

#### EDT-V3-04 — Single-letter shortcuts surfaced in toolbar tooltips

**Status.** `now`. R-A "What we'd copy" #2.

**Scope.** Every toolbar button shows its single-letter shortcut as a chip in the tooltip AND as an 8 px chip in the bottom-right corner of the icon (always visible, no hover). Shortcuts are already wired in code — this WP surfaces them in chrome.

**Data model.** Each tool registration gains a `shortcut: string` field used for both keyboard binding and chip label.

**UI.** 8 × 8 px chip in `--text-2xs`, `--color-muted-foreground` on `--surface-2`, bottom-right corner of icon. Tooltip: "Wall (W)" with W chip.

**Acceptance.** Wall tool tooltip reads "Wall (W)" with `W` chip permanently visible on icon. Door/Window/Dimension/Phase show `D`/`WI`/`DI`/`PH`. Visual regression confirms placement.

**Effort.** S — 3 days.

## Implementation tasks

### 1. Audit existing tool registrations

Find all toolbar tool definitions — they may be in `PlanToolbar.tsx`, a `tools.ts` config file, or similar. List the tools and their current keyboard shortcuts. At minimum these tools must have shortcut chips:

| Tool | Shortcut |
|------|---------|
| Wall | `W` |
| Door | `D` |
| Window | `WI` |
| Dimension | `DI` |
| Phase | `PH` |
| Stair | `S` |
| Roof | `R` |
| Column | `C` |
| Detail Region | `DR` |

Add a `shortcut?: string` field to the tool definition type if not already present.

### 2. `ShortcutChip` component (`packages/web/src/ui/ShortcutChip.tsx`)

```tsx
export function ShortcutChip({ label }: { label: string }) {
  return (
    <span
      style={{
        position: 'absolute',
        bottom: 2,
        right: 2,
        fontSize: 'var(--text-2xs)',
        color: 'var(--color-muted-foreground)',
        background: 'var(--surface-2)',
        padding: '0 2px',
        borderRadius: 2,
        lineHeight: 1,
        pointerEvents: 'none',
        userSelect: 'none',
      }}
    >
      {label}
    </span>
  );
}
```

No hex literals — tokens only.

### 3. Wire `ShortcutChip` into toolbar buttons

In the toolbar button component (wherever `PlanToolbar.tsx` renders individual tool buttons):

- Wrap the icon container in `position: relative`
- Render `<ShortcutChip label={tool.shortcut} />` inside if `tool.shortcut` is defined
- Update the tooltip text from `"Wall"` to `"Wall (W)"` when shortcut is present

Pattern:
```tsx
<button
  style={{ position: 'relative' }}
  title={tool.shortcut ? `${tool.label} (${tool.shortcut})` : tool.label}
  onClick={() => activateTool(tool.id)}
>
  <ToolIcon id={tool.id} />
  {tool.shortcut && <ShortcutChip label={tool.shortcut} />}
</button>
```

### 4. Keyboard binding audit

Verify the keyboard shortcuts listed above are actually bound. If a shortcut is in the tool definition but not wired to a `keydown` handler, add the binding in the existing keyboard handler (usually in a `useToolKeyBindings` hook or similar).

Two-letter shortcuts (`WI`, `DI`, `PH`, `DR`) use a chord: within 400 ms of pressing the first key, pressing the second activates the tool. If no chord infrastructure exists, add a simple `pendingChord: string | null` state in the keyboard handler.

### 5. Modifier bar shortcut display

If `packages/web/src/tools/modifierBar.ts` exposes modifier keys (e.g. `L` for loop mode), ensure those also follow the chip pattern when rendered in the OptionsBar.

### 6. Tests

No Python tests. Add a lightweight snapshot or unit test confirming:

- `ShortcutChip` renders with the correct label
- Tool definitions for Wall/Door/Window include a `shortcut` field
- Tooltip string includes `(W)` suffix when shortcut is present

## Implementation rules

- **No scope creep.** Do not wire loop-mode modifier here (EDT-V3-05). Do not add new keyboard shortcuts beyond auditing existing ones.
- **Chips are always visible** — no hover required. D3: every glyph carries meaning.
- **No hex literals** in any `packages/web/src/**/*.tsx`.
- A5 antidote: keyboard is discoverable by reading the toolbar, not by hunting tooltips.
