## Implementation tasks

### 1. Inventory the icon set

Run:
```bash
ls /Users/jhoetter/repos/bim-ai/packages/icons/src/*.tsx | xargs grep -l "^export"
```

Open `packages/icons/src/index.ts` to see all exported icon names. Build a mental map of what's available before editing toolbars.

Key icon categories available (read the actual files to confirm names):

- `views.tsx` — view type icons
- `edit.tsx` — edit actions
- `workflow.tsx` — general workflow actions
- `collaboration.tsx` — collab icons
- `organization.tsx` — organisation/nav icons

### 2. Audit toolbar components

Run a search for text-only buttons and emoji patterns:

```bash
# Find buttons with text content but no icon import
grep -rn "☀\|✓\|×\|✕\|→\|←\|⊕\|◇\|○" packages/web/src/workspace/ packages/web/src/plan/ \
  --include="*.tsx" | grep -v ".test\."

# Find buttons/spans that likely have text labels without icons
grep -rn "<button[^>]*>[A-Z][a-z]" packages/web/src/workspace/ packages/web/src/plan/ \
  --include="*.tsx" | grep -v ".test\.\|//\|{t(" | head -30
```

Document what you find: a list of (file, line, current content, proposed replacement).

### 3. Replace emoji and ASCII with proper icons or remove

For each emoji/ASCII stand-in found:

- `☀` (sun): This should be removed from the `SunOverlay` button (which VIS-V3-04 handles), but check if it appears elsewhere. If still present and VIS-V3-04 is not yet merged, replace with `Icons.sun` if it exists, or remove the emoji and use the button label only with a `title`.
- `✓` (check/confirm): Replace with an existing check/confirm icon from `edit.tsx` or `workflow.tsx`.
- `×` / `✕` (close/delete): Replace with `Icons.close` or similar from the icon set.
- `→` / `←` (nav): Replace with navigation icons if available.

For each replacement: import the icon from `@bim-ai/icons` and render it at the appropriate `size` (typically `ICON_SIZE.chrome` = 16 or the constant used in the file).

### 4. Replace text-label-only toolbar buttons with icon + tooltip

For each toolbar button that has text but no icon:

1. Find the matching icon in `@bim-ai/icons` (e.g. "Duplicate" → check for `DuplicateIcon` in `edit.tsx`).
2. Replace `<button>Duplicate</button>` with `<button title="Duplicate"><Icons.duplicate size={ICON_SIZE.chrome} aria-hidden="true" /></button>`.
3. If the button currently shows text alongside an icon, and the panel is in a toolbar context (not an expanded rail at ≥ 240 px), remove the text and keep only the icon.

**A5 rule:** Text labels are allowed — and preferred — at panel widths ≥ 240 px. Icon-only is appropriate for compact toolbars, the status bar, and the collapsed left rail. Do not blindly remove all text labels — only those in dense toolbar/icon-strip contexts.

### 5. Confirm `title` attributes

After replacements, confirm every icon-only button has a `title` attribute. This is both an accessibility requirement and the hover tooltip source.

Run:
```bash
grep -rn "<button[^>]*>" packages/web/src/workspace/ packages/web/src/plan/ \
  --include="*.tsx" | grep -v "title=\|{t(\|test" | grep -v "//"
```

Any button in the result without a `title` (and without visible text content) is a gap — add `title`.

## Implementation rules

- **Use only existing icons.** Do not author new SVG icons in this WP. If a semantic icon is missing, note it in the final report for a follow-up (the next icon set expansion), and leave the text label in place.
- **A5 rule preserved.** In expanded rail or panel contexts (≥ 240 px width), text labels alongside icons are the right pattern — do not remove them. Only remove text from compact/dense toolbars.
- **No hex literals.** Icon components render `currentColor` — no colour changes needed.
- **No scope creep.** Do not redesign toolbar layout, add new toolbar sections, or change icon sizes beyond what the existing `ICON_SIZE.*` constants specify.
