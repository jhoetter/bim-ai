## Verbatim MRK-V3-03 tracker entry (§6 T3, line 1507)

#### MRK-V3-03 — Sheet review surface with comment-anchoring

_Source: R-B §9 #1 (sheet review with bidirectional Revit comments — the email/PDF chain practices universally hate). Severity High. The Bluebeam-replacement WP._

**Status.** `next`.
**Scope.** Published sheets render in a dedicated review surface inside the canvas. Reviewers (including public-link viewers) can drop comments + markups on sheet detail callouts; the comment back-flows to the source view from which the detail was sliced. Comment chip appears in the source view's chrome: "1 comment on detail B-12 from sheet A-301." Click → jumps to source view with the comment surfaced as a pin. Removes the email/PDF chain that practices universally hate.
**Data model.** Reuses MRK-V3-01 `Comment` entity with extended anchor:
```ts
anchor:
  | …
  | { kind: 'sheet'; sheetId: string; xPx: number; yPx: number; sourceViewId?: string; sourceElementId?: string };
```
**Engine.** Sheet anchor includes optional `sourceViewId` + `sourceElementId` populated by the sheet renderer (which knows which view and element each pixel maps from). When the comment is created with a sheet anchor, server resolves the source binding and surfaces a notification chip on the source view. Bidirectional: closing the comment in either surface (sheet or source view) closes both.
**UI.** Sheet review surface is a dedicated full-screen surface (not a modal) — sheet renders at print fidelity, review toolbar at the bottom (`CM` comment, `AN` markup, `MR` resolve). Source-view chip: small flag glyph in the view header with a count; click opens a panel listing sheet-originated comments.
**Acceptance.** Publish a sheet with detail B-12 sliced from plan view P-1. Reviewer opens the public link, drops a comment on detail B-12. Architect, working in plan view P-1, sees a chip "1 comment from sheet A-301" within 2s. Click → comment surfaces as a pin on the actual element in P-1. Resolve from P-1; the sheet-side comment also resolves.
**Effort.** L — 2.5 weeks (depends on T4's sheet rendering being mature enough to expose source-element bindings).
**Cross-theme references.** T4 (sheets + source-element bindings); MRK-V3-01 (Comment entity); T9 (reviewer-suggested edits on sheets can be picked up by the agent as a refinement prompt).

## Implementation tasks

### 1. Extend `SheetAnchor` in Python (`app/bim_ai/comments.py`)

Add a new anchor variant alongside `ElementAnchor`, `PointAnchor`, `RegionAnchor`:

```python
class SheetAnchor(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    kind: Literal["sheet"] = "sheet"
    sheet_id: str = Field(alias="sheetId")
    x_px: float = Field(alias="xPx")
    y_px: float = Field(alias="yPx")
    source_view_id: str | None = Field(default=None, alias="sourceViewId")
    source_element_id: str | None = Field(default=None, alias="sourceElementId")
```

Extend the `CommentAnchor` discriminated union to include `SheetAnchor`:

```python
CommentAnchor = Annotated[
    ElementAnchor | PointAnchor | RegionAnchor | SheetAnchor,
    Field(discriminator="kind"),
]
```

No other changes to `Comment` — the entity is unchanged; only the anchor union grows.

### 2. Source-binding resolution (`app/bim_ai/comments.py`)

Add a helper called at comment-creation time when the anchor kind is `"sheet"`:

```python
def resolve_sheet_anchor_binding(
    anchor: SheetAnchor,
    sheet_pixel_map: dict[tuple[int, int], tuple[str, str]],
    # maps (x_px_rounded, y_px_rounded) → (sourceViewId, sourceElementId)
) -> SheetAnchor:
    """Populate sourceViewId + sourceElementId from the sheet pixel map if available."""
    key = (round(anchor.x_px), round(anchor.y_px))
    binding = sheet_pixel_map.get(key)
    if binding:
        return anchor.model_copy(update={"source_view_id": binding[0], "source_element_id": binding[1]})
    return anchor
```

The pixel map is provided by the sheet renderer at request time. For v3, the map is returned as a JSON sidecar by `GET /api/v3/sheets/{sheetId}/pixel-map` (see task 3). If no map entry exists for the clicked pixel, the anchor stores `None` for both optional fields — the comment is still valid, just without back-flow.

### 3. New REST endpoint: pixel map (`app/bim_ai/routes_api.py`)

```
GET /api/v3/models/{modelId}/sheets/{sheetId}/pixel-map
```

Returns a JSON object `{ "map": { "<x>,<y>": { "sourceViewId": "…", "sourceElementId": "…" } } }` for integer pixel coordinates. The server-side sheet renderer populates this by walking `sheet.viewPlacements`, converting each placement's bounding box to pixel ranges, and recording the viewId + element under each pixel from the source view's hit-test index. For v3, approximate coverage is sufficient — all pixels in a view-placement bounding box map to that viewId; sourceElementId is populated only where a hit-test index exists.

Register the endpoint under the existing `router` in `routes_api.py`. Requires `viewer` or above permission (public-link viewers included).

### 4. Notification chip on source view (`app/bim_ai/routes_api.py` + `app/bim_ai/activity.py`)

When a `POST /api/v3/models/{modelId}/comments` lands with `anchor.kind == "sheet"` and `anchor.sourceViewId` is not `None`:

- Write a notification entry to the activity stream for the source view: `{ kind: "sheet_comment_chip", viewId: sourceViewId, sheetId: sheetId, commentId: commentId, sheetNumber: … }`.
- The activity stream already broadcasts via WebSocket; the chip appears within 2 s for connected clients (CQ-01 prerequisite — assumed done).

Add `SheetCommentChipEntry` to `activity.py` alongside existing activity entry types.

### 5. Bidirectional resolve (`app/bim_ai/routes_api.py`)

When a comment with a `sheet` anchor is resolved (via `POST /api/v3/models/{modelId}/comments/{commentId}/resolve`), also mark resolved any comment in the same `threadId` that has a matching `sourceElementId` anchor on the source view (element anchor pointing at `sourceElementId`). Symmetric: resolving from the element-anchor side resolves the sheet-side thread.

Implement as a post-resolve hook in the existing resolve handler. This is a best-effort match — if no matching thread exists on the element side, resolve proceeds silently.

### 7. `SheetReviewSurface` component (`packages/web/src/plan/SheetReviewSurface.tsx`)

Full-screen review surface (not a modal — anti-pattern A4 antidote):

```tsx
type SheetReviewSurfaceProps = {
  sheetId: string;
  modelId: string;
  readOnly?: boolean; // true for public-link viewers without editor role
};
```

- Renders `<SheetCanvas>` at print fidelity occupying the full viewport minus the 40 px bottom toolbar.
- Bottom review toolbar (40 px, `--color-surface-strong`, `--radius-canvas: 0`): three shortcut chips — `CM` comment, `AN` markup, `MR` resolve — matching the existing toolbar pattern from `MarkupToolbar.tsx`.
- In `CM` mode, a click on the sheet canvas: (a) fetches the pixel map (cached per sheetId, 30 s TTL); (b) resolves `sourceViewId` + `sourceElementId`; (c) opens a `CommentsPanel` thread popover at `--space-2` from the pin; (d) on post, dispatches `POST /comments` with a `sheet` anchor.
- Comment pins on the sheet render as `<CommentPin>` with `kind="sheet"` variant — same SDF glyph at constant screen size, no hex literals.
- Uses `var(--color-surface-1)` for the sheet background pass-through.

### 8. `SourceViewChip` component (`packages/web/src/workspace/SourceViewChip.tsx`)

Small chip in the view header (plan view, 3D view) surfacing sheet-originated comments:


- Reads from the activity stream (WebSocket) for `sheet_comment_chip` entries matching `viewId`.

- Zero state: chip is hidden (`display: none`) — D1 / A9 antidote.
- No hex literals. Tokens only.

## Implementation rules

- **Agent-callable contract per API-V3-01.** Every new route must have: HTTP endpoint registered in `routes_api.py`, Pydantic request/response schemas, and a JSON schema entry in `app/bim_ai/commands.py` if it mutates state. CLI exposure is not required for this WP (the sheet review surface is UI-driven), but the REST contract must be stable.
- **No scope creep.** Do not author BCF export, IFC bindings, or any T9 agent tooling in this WP. Do not ship live-cursor overlays on the sheet surface (COL-V3-04, `vision`). Do not add a chat sidebar.
- **Design-pillar adherence.** Sheet review surface is full-screen, never a modal (A4 antidote). Bottom toolbar shortcut chips carry text labels at widths ≥ 240 px (A5 antidote). `SourceViewChip` is hidden when count is zero (A9 antidote). Resolve from either surface; no separate "synch" step (A6 antidote). D6 load-bearing: comment threads are first-class, threadable, resolvable, non-modal.
- **No hex literals in `packages/web/src/**/*.tsx`.** The ESLint rule (`bim-ai/no-hex-in-chrome`) will fail CI. Use `var(--color-*)`, `var(--draft-*)`, `var(--disc-*)` exclusively. If a token is missing, add it to `tokens-v3.css` — do not hardcode values.
