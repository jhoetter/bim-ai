### 2. Migrate `wallTypeCatalog.ts`

In `packages/web/src/families/wallTypeCatalog.ts`, replace every hex literal with a `readToken('--mat-<name>', '<hex-fallback>')` call. The fallback hex is only for SSR safety and must match the token value. Example:

```ts
// Before:
fill: '#7c5b3b',

// After:
fill: readToken('--mat-timber-cedar', '#7c5b3b'),
```

If `readToken` doesn't exist in the codebase, implement it as:

```ts
function readToken(varName: string, fallback: string): string {
  if (typeof window === 'undefined') return fallback;
  return getComputedStyle(document.documentElement).getPropertyValue(varName).trim() || fallback;
}
```

Place this function in `packages/web/src/styles/tokens.ts` (create if needed) and import it.

### 3. ESLint rule (optional but preferred)

If the codebase has a `bim-ai/no-hex-in-chrome` ESLint rule already configured, remove `wallTypeCatalog.ts` from its allowlist. If not, add a comment at the top of `wallTypeCatalog.ts`:

```ts
/* eslint-disable bim-ai/no-hex-in-chrome -- migrated to --mat-* tokens; no hex literals below */
```

and ensure the file passes lint with that rule enabled. Skip if the ESLint rule doesn't exist — do not create the rule from scratch (that is CQ scope).

### 4. No visual regression

The wall fills in plan and 3D views must look byte-identical to before. Since the token values match the old hex literals exactly, this is guaranteed. Confirm by running the existing visual snapshot tests (if any) and `pnpm exec tsc --noEmit`.
