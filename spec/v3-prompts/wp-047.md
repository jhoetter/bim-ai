## WP entry (verbatim from tracker)

```
#### CHR-V3-08 — Secondary contextual bar

**Status.** `next`. Cross-ref T2 EDT-V3-04 + EDT-V3-05.
**Scope.** 36 px secondary bar morphs by active tool. Every authoring tool ships a
`<ToolModifierBar>` mount with the same vocabulary (Place / Chain / Multiple / Loop /
Tag-on-Place / Numeric / Tab-cycle).
**Data model.** New `ToolModifierDescriptor` interface in
`packages/web/src/tools/modifierBar.ts`.
**Engine.** Modifier flips dispatch tool-state actions; sticky-per-session via zustand
`toolPrefs`.
**UI.** `--text-xs` modifier labels with shortcut chips. `--surface-2` background, 1 px
`--color-border` bottom.
**Acceptance.** Wall → alignment cycle (`S`), Loop (`L`), Multiple, Tag-on-Place.
Door → swing-side cycle. Same vocabulary, different content.
**Effort.** M.
```

Modifier vocabulary (from tracker §UX, the table CHR-V3-08 codifies):

| Modifier      | Meaning                                          | Default           |
|---------------|--------------------------------------------------|-------------------|
| Place         | One-shot placement                               | on (wall default) |
| Chain         | Segment-chained drawing                          | on (wall default) |
| Multiple      | Place repeatedly without re-pressing tool key    | off               |
| Loop (L)      | Auto-restart after each completed segment        | off; sticky/session|
| Tag-on-Place  | Place a tag/dimension at create time             | off               |
| Numeric       | Type to override value mid-draw                  | always armed      |
| Tab-cycle     | Tab moves between numeric fields                 | always armed      |

## Implementation rules (verbatim from §A)

- **Honour API-V3-01.** Every kernel command you add must have CLI + REST/RPC + JSON schema
  + assumption-log support. Add tests at the agent-callable layer (`tests/cli/`,
  `tests/api/`). Modifier flips that mutate session-level `toolPrefs` must be commandable
  and logged.
- **No scope creep.** Do not silently add features not in this WP entry. EDT-V3-04
  (shortcut chips on toolbar icons), EDT-V3-05 (loop-mode mid-command modifier), and
  EDT-V3-12 (numeric-override polish) are separate WPs — do not reimplement them here.
  CHR-V3-08's job is the shared `<ToolModifierBar>` mount and `ToolModifierDescriptor`
  contract; per-tool modifier content is defined by each tool's own registration.
- **Design-pillar adherence.** This WP directly serves D3 (density-conscious, never crowded),
  D8 (context, not configuration), and is the antidote to A3 (ribbon-with-30-tabs). The bar
  must morph by active tool — no fixed set of tabs, no modal, no dialog (A4 antidote). The
  modifier bar is not a second toolbar; it is the context layer beneath the primary tool verb.
- **No hex literals in `packages/web/src/**/*.tsx`.** All colours in the modifier bar must
  consume CSS custom properties: `var(--surface-2)`, `var(--color-border)`,
  `var(--text-xs)`, `var(--color-muted-foreground)`, etc. The ESLint rule (R-G §2.5 phase 3)
  will fail the build otherwise. Do not introduce any raw hex string in any `.tsx` file
  under `packages/web/src/`.
- **Sticky-per-session via zustand `toolPrefs`.** Modifier state (Loop on/off, Multiple
  on/off, Tag-on-Place on/off) must persist to the `toolPrefs` zustand slice for the
  session-default UX. Tool switching should restore the last-used modifier state for that
  tool.
- **Same vocabulary, different content.** Every authoring tool (Wall, Door, Window, Column,
  Beam, Stair, Roof, Dimension, Section) mounts `<ToolModifierBar>` with its own
  `ToolModifierDescriptor[]`. The bar's layout, animation, and token usage are identical
  across all tools — only the descriptors differ.
- **36 px height, 200 ms `--ease-paper` mount.** The bar appears when a tool is active and
  disappears (not hides) when no tool is active. It does not persist between tool sessions
  (D8 — context, not configuration).
