## Implementation tasks

### 1. Define the workspace catalog

Create `packages/web/src/workspace/chrome/workspaces.ts`. This is a pure static module — no imports from the store.

```ts
export type WorkspaceId = 'arch' | 'struct' | 'mep';

export interface WorkspaceDescriptor {
  id: WorkspaceId;
  label: string;           // display name
  discToken: string;       // e.g. 'var(--disc-arch)'
  discSoftToken: string;   // e.g. 'var(--disc-arch-soft)'
  defaultLensLabel: string;
}

export const WORKSPACES: WorkspaceDescriptor[] = [
  {
    id: 'arch',
    label: 'Architekt',
    discToken: 'var(--disc-arch)',
    discSoftToken: 'var(--disc-arch-soft)',
    defaultLensLabel: 'All',
  },
  {
    id: 'struct',
    label: 'Statiker',
    discToken: 'var(--disc-struct)',
    discSoftToken: 'var(--disc-struct-soft)',
    defaultLensLabel: 'Structure',
  },
  {
    id: 'mep',
    label: 'TGA',
    discToken: 'var(--disc-mep)',
    discSoftToken: 'var(--disc-mep-soft)',
    defaultLensLabel: 'MEP',
  },

```

Do not hardcode `#6e8a72` or any hex. Every color value is a CSS variable string that the browser resolves at paint time.

The component is **purely presentational**. It does not import from any store. The caller (AppShell or the Workspace container) passes `onSetActiveWorkspace` which wraps the `SetActiveWorkspace` dispatch.

### 3. Chip — closed state

The chip is a `<button type="button">` with:

- `data-testid="workspace-switcher-chip"`
- `data-disc={activeWorkspaceId}` — used by acceptance tests and LNS-V3-02 chrome tinting
- `aria-haspopup="listbox"`
- `aria-expanded={isOpen}`

Visual specification:
- Width: `auto`; height: `var(--shell-topbar-height)` minus 16 px (i.e. centred vertically)
- Left border: `4px solid` in `var(--disc-*)` for the active workspace — **not a background**, a border
- Background closed: `var(--color-surface)`, hover `var(--color-surface-strong)`
- Typography: `var(--text-sm)`, `var(--text-sm-line)`, medium weight
- Disclosure glyph: `▾` at `var(--color-muted-foreground)`
- `border-radius: var(--radius-md)` on the chip body (interior interactive element)
- `transition: background-color var(--motion-slow) var(--ease-paper)` — the 240 ms flip lives here

```tsx
const chipStyle: React.CSSProperties = {
  display: 'flex',
  alignItems: 'center',
  gap: 6,
  padding: '0 8px',
  borderLeft: `4px solid ${activeWorkspace.discToken}`,
  borderTop: 'none',
  borderRight: 'none',
  borderBottom: 'none',
  borderRadius: 'var(--radius-md)',
  backgroundColor: 'var(--color-surface)',
  fontSize: 'var(--text-sm)',
  lineHeight: 'var(--text-sm-line)',

```

### 4. Dropdown menu — open state

Use a `<ul role="listbox">` positioned absolutely below the chip. Manage open/close with local `useState`. Close on outside click (`useEffect` + `mousedown` on `document`) and on `Escape` keydown.

Menu container style:
- `background: var(--color-surface)`
- `border: 1px solid var(--color-border)`
- `border-radius: var(--radius-md)`
- `box-shadow: var(--elev-2)`
- `min-width: 160px`
- `z-index: 100`
- Animation: `opacity 0 → 1` + `translateY(-4px) → translateY(0)` in `var(--motion-slow)` `var(--ease-paper)`

### 5. Dropdown rows — 4 entries

Rows for `arch`, `struct`, `mep`, plus a divider then a fourth row for **Concept** (T6 — render as disabled/`aria-disabled="true"` in this WP; CHR-V3-02 is a T6 cousin, full Concept workspace ships in T6).

Each row (`<li role="option">`):

- Left swatch: `8px × 8px` circle, `background: var(--disc-*)` for that workspace, `border-radius: var(--radius-pill)`
- Label: `var(--text-sm)`, `var(--color-foreground)`
- Active indicator: checkmark icon on the right when `id === activeWorkspaceId`
- Hover: `background: var(--color-surface-strong)`
- `data-testid={`workspace-option-${id}`}`
- `aria-selected={id === activeWorkspaceId}`

```tsx
const rowStyle = (isActive: boolean, discToken: string): React.CSSProperties => ({
  display: 'flex',
  alignItems: 'center',
  gap: 8,
  padding: '6px 10px',
  cursor: 'pointer',

  fontSize: 'var(--text-sm)',
  lineHeight: 'var(--text-sm-line)',
  color: 'var(--color-foreground)',
});
```

No hex. The swatch circle uses `backgroundColor: discToken` where `discToken` is `'var(--disc-arch)'` etc.

### 6. Selection transition — 240 ms `--ease-paper`

When `onSetActiveWorkspace(id)` fires:
1. The menu closes immediately.
2. The chip's left-border color transitions to the new `--disc-*` value in 240 ms via the `transition` rule already declared on the chip (step 3 above).
3. The `data-disc` attribute on the chip updates synchronously — this is what `LNS-V3-02` observes to flip the chrome tint.

No JS animation library needed. The CSS `transition` on `border-color` handles the visual flip. The caller (`AppShell`) is responsible for propagating the new `activeWorkspaceId` down as a prop change — the component itself has no internal "previous state" tween.

The tracker acceptance test reads: "Architekt → Statiker flips stripe, re-orders left-rail stack, preselects status-bar lens to 'Structure' in 240 ms." The 240 ms is the CSS transition; the rail re-order and lens preselect are side-effects owned by the store (the component fires `onSetActiveWorkspace`; the store handles the cascade). **This component is not responsible for the rail reorder or the lens preselect** — it fires the command and trusts the store.

## Implementation rules

1. **No hex literals anywhere.** `bim-ai/no-hex-in-chrome` ESLint rule is enforced in CI. Every color is a CSS variable: `var(--disc-arch)`, `var(--disc-struct)`, `var(--disc-mep)`, `var(--color-surface)`, `var(--color-border)`, `var(--brand-accent)`. The only strings that may look like colors are CSS variable references. **Violations fail the brand-swap CI invariant test** — when the brand layer is swapped, the structural tokens (`--disc-*`) must remain byte-identical; a hex literal breaks that guarantee.
2. **`--ease-paper` is the only motion easing permitted.** Duration: `var(--motion-slow)` (240 ms). No `ease`, `ease-in-out`, `linear`, or numeric cubic-bezier literals.
3. **`--radius-canvas: 0` on rail/bar surfaces; `--radius-md` on interior chips and menus.** The chip and menu are interior interactive elements; they may use `--radius-md`. The parent top-bar root may not.
4. **Pure presentational component.** No store imports in `WorkspaceSwitcher.tsx` or `workspaces.ts`. State is passed as props. The `onSetActiveWorkspace` callback is the only side-effect boundary.
5. **`--text-sm` throughout.** All labels, row text, and disclosure glyphs use `var(--text-sm)` / `var(--text-sm-line)`. No other type sizes in this component.
6. **Do not implement the Concept workspace.** Render the row as `aria-disabled="true"` with a `title="Coming in T6"` tooltip. The full Concept workspace is T6 scope.
7. **The `data-disc` attribute is a contract.** LNS-V3-02 (chrome discipline tinting) reads `[data-disc]` on the chip to apply the chrome tint. Do not rename it.
8. **Concept row uses `--disc-arch-soft` swatch.** The Concept entry does not have its own `--disc-concept` token in Sprint 0. Use `var(--disc-arch-soft)` as a placeholder until T6 ships a dedicated token.
9. **Export both the component and the catalog.** `packages/web/src/workspace/chrome/WorkspaceSwitcher.tsx` exports `WorkspaceSwitcher` and `WorkspaceSwitcherProps`. `packages/web/src/workspace/chrome/workspaces.ts` exports `WORKSPACES`, `WorkspaceDescriptor`, `WorkspaceId`.
10. **Do not touch anything outside `packages/web/src/workspace/chrome/`.** The only files modified outside that directory are `chrome/TopBar.tsx` (stub replacement) and the new test file.
