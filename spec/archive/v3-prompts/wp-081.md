## Implementation tasks

### 1. Understand the current wiring

Open `packages/web/src/Viewport.tsx`. Search for `SunOverlay`. The component is imported around line 94 and rendered around line 1990. It receives:
- `values={sunOverlayValues}`
- `azimuthDeg={sunAzimuth}`
- `elevationDeg={sunElevation}`
- `onChange={handleSunChange}`
- `onCommit={handleSunCommit}`

The `sunOverlayValues` state, `setSunOverlayValues`, and the `handleSunChange`/`handleSunCommit` callbacks currently all live in `Viewport.tsx`. They need to be lifted to the parent (`Workspace.tsx` or its equivalent) so they can be shared with the new inspector panel without creating a prop-drilling chain.

### 2. Lift sun state out of `Viewport.tsx`

**Option A (recommended):** Add a `sunStore` zustand slice in `packages/web/src/sunStore.ts`:

```typescript
import { create } from 'zustand';
import type { SunOverlayValues } from './viewport/SunOverlay';

type SunStore = {
  values: SunOverlayValues;
  azimuthDeg: number;
  elevationDeg: number;
  setValues: (patch: Partial<SunOverlayValues>) => void;
  setComputedPosition: (azimuth: number, elevation: number) => void;
};

const DEFAULT_VALUES: SunOverlayValues = {
  latitudeDeg: 48.14,
  longitudeDeg: 11.58,
  dateIso: new Date().toISOString().slice(0, 10),
  hours: 14,
  minutes: 0,
  daylightSavingStrategy: 'auto',
};

export const useSunStore = create<SunStore>((set) => ({
  values: DEFAULT_VALUES,
  azimuthDeg: 145,
  elevationDeg: 35,
  setValues: (patch) => set((s) => ({ values: { ...s.values, ...patch } })),
  setComputedPosition: (azimuthDeg, elevationDeg) => set({ azimuthDeg, elevationDeg }),
}));
```

In `Viewport.tsx`: replace `const [sunOverlayValues, setSunOverlayValues] = useState(...)` with `const { values: sunOverlayValues, setValues: setSunOverlayValues } = useSunStore()`. Remove the `SunOverlay` import and render from `Viewport.tsx`. The NOAA computation logic (`useEffect` that calls `computeSunPositionNoaa`) stays in `Viewport.tsx` — it calls `useSunStore.getState().setComputedPosition()`.

### 3. Create `SunInspectorPanel.tsx`

Create `packages/web/src/workspace/SunInspectorPanel.tsx`. This component renders the sun controls in a collapsible inspector section:

```tsx
// SunInspectorPanel — sun & shadow controls for the right-rail Scene section.
// Replaces the floating SunOverlay. No hex literals.
import { useSunStore } from '../sunStore';
import { computeSunPositionNoaa } from '../viewport/sunPositionNoaa';

const SEASON_PRESETS = [
  { label: 'Winter solstice', month: 12, day: 21 },
  { label: 'Equinox', month: 3, day: 20 },
  { label: 'Summer solstice', month: 6, day: 21 },
] as const;
```

**Sun arc SVG widget spec:**

Render a `120 × 60 px` half-circle representing the sky hemisphere. A draggable `<circle>` dot (r=5) indicates the sun position:
- Horizontal position (x) maps to azimuth: `x = 60 + 55 * cos(Math.PI - azimuthRad)` where `azimuthRad` is azimuth in radians scaled to 0–π across the full 360° azimuth range. (Simplify: map azimuth 0–360 linearly to x 5–115.)
- Vertical position (y) maps to elevation: `y = 55 - (elevation / 90) * 50` (elevation 0° at bottom, 90° at top).
- Drag interaction: on `onMouseDown` / `onPointerDown`, compute new azimuth from x and elevation from y. Call `setValues` with the corresponding lat/lon/time that would produce those angles, OR simply call `onCommit` with a direct `azimuthDeg`/`elevationDeg` override (requires extending `SunOverlayValues` with optional override fields — see implementation note below).
- The arc: `<path d="M 5 55 A 55 55 0 0 1 115 55" stroke="var(--color-border)" fill="none"/>`. The ground line: `<line x1="5" y1="55" x2="115" y2="55" stroke="var(--color-border-strong)"/>`.

**Implementation note on drag:** The cleanest approach is to let the drag set `hours` and `minutes` in the sun store (since azimuth/elevation are computed from lat/lon/date/time). Map the drag x-position to an hour of day (5am → 7pm = x=5→115). This gives a natural "slide the sun across the sky" interaction that users understand. The elevation follows the NOAA computation automatically.

**Season presets:** Three buttons ("Winter", "Equinox", "Summer") that set `dateIso` to the corresponding solstice/equinox date in the current year. Below these, an optional `<input type="date">` for a specific date.

**Advanced section (collapsed by default):**
```tsx
<details>
  <summary style={{ fontSize: 'var(--text-xs)', color: 'var(--color-muted-foreground)', cursor: 'pointer' }}>
    Advanced
  </summary>
  {/* Latitude, longitude, daylight saving inputs */}
</details>
```

Move the lat/lon numeric inputs and DST selector into this collapsible. They were always engineering detail — hide them by default.

**Status readout (always visible, above the arc):**
```
"14 Jun, 14:00 — sun at 145°, 35° elevation"
```
One line of `--text-xs` `--color-muted-foreground`.

## Implementation rules

- **No floating sun overlay on the canvas.** After this WP, `Viewport.tsx` must not render any `position: absolute` or `position: fixed` sun controls. The canvas surface should contain only the 3D scene.
- **No hex literals.** All colours via CSS custom properties. The SVG arc and ground line use `var(--color-border)`, dot uses `var(--color-accent)`.
- **No scope creep.** Do not redesign the inspector or add other scene controls in this WP. Only sun + shadow belong here.
- **`SunOverlayValues` type is stable.** Do not remove or rename fields — callers depend on this type.
