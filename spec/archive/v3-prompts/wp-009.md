## Verbatim CMD-V3-01 tracker entry (§6 T9, line 3136)

**CMD-V3-01 — Command-bundle apply API**

_Source: User reframe 2026-05-08 ("the AI features are designed as API / CLI tools"); §L Locked Decisions (deterministic kernel boundary). Reframes the previous "agent edit loop" WP into a kernel API — the loop logic moves out to the agent; we ship the deterministic apply primitive._

**Status.** `now`. **Scope.** A deterministic kernel API + CLI where any caller submits a `CommandBundle{ commands, assumptions, parent_revision, targetOptionId? }` and gets back `BundleResult{ applied, violations, new_revision }`. The "loop" — propose → dry-run → read advisories → repair → commit — is what an _agent_ does using this API. We ship the API, not the loop. This is the cleanest expression of §L: bim-ai's binary contains zero AI; the agent composes calls using its own reasoning.

Three call modes: (1) `--dry-run` — runs SKB-05 + SKB-11 + SKB-19 + soundness pack against a hypothetical post-bundle state; returns advisories + checkpoint snapshot id; **does not mutate**. Idempotent. (2) `--commit` — same validation, then commits if no blocking advisories fire; returns new revision id; on failure, returns advisories and the bundle is _not_ applied. (3) `--commit --tolerate <advisory-class>` — explicit override; each tolerance recorded in audit log.

Every bundle MUST include a non-empty `assumptions` field (CMD-V3-02 contract), `parent_revision`, and target a Design Option that is not `main` (OPT-V3-01 invariant). The kernel rejects violators with structured advisories.

**Data model.**
```ts
type CommandBundle = {
  schemaVersion: 'cmd-v3.0';
  commands: KernelCommand[];
  assumptions: AssumptionEntry[];            // CMD-V3-02 — non-empty
  parentRevision: string;                    // optimistic concurrency
  targetOptionId?: string;                   // OPT-V3-01 — defaults to a fresh option
  tolerances?: Array<{ advisoryClass: string; reason: string }>;
};
type BundleResult = {
  schemaVersion: 'cmd-v3.0';
  applied: boolean;
  newRevision?: string;
  optionId?: string;
  violations: Advisory[];
  checkpointSnapshotId?: string;             // hand-off to VG-V3-01
};
```

**Engine.** New module `app/bim_ai/cmd/` with `apply_bundle(bundle, mode) → BundleResult`. Wraps kernel command application with the validation + assumption + revision contract. CLI: `bim-ai apply-bundle <bundle.json> --base <revision> [--dry-run | --commit]`. REST: `POST /api/v3/bundles`. Both registered per API-V3-01. Idempotent: same bundle replayed against the same parent yields the same new revision. **UI.** No direct UI. **Acceptance.** (a) Dry-run returns advisories + checkpoint id; kernel state unchanged. (b) `--commit` produces new revision + new option id; `main` untouched. (c) Bundle missing `assumptions` rejected with `assumption_log_required`. (d) Stale `parent_revision` rejected with `revision_conflict`. (e) Idempotent replay returns existing revision id with `applied: false`. (f) An external agent using only `bim-ai apply-bundle` + `bim-ai compare` can complete the 12-step refinement fixture (TST-V3-01). **Effort.** XL — 4 weeks. **Cross-theme references.** API-V3-01. T1 KRN-V3-04 (every commit lands as an option per OPT-V3-01). T5 (catalogKey resolution at apply time). CQ-04 (`engine.py` split must complete first).

## Implementation tasks

### 1. `CommandBundle` + `BundleResult` Pydantic types — `app/bim_ai/cmd/types.py`

- `CommandBundle`: `schema_version: Literal["cmd-v3.0"]`, `commands: list[dict]`, `assumptions: list[AssumptionEntry]` (`min_length=1`), `parent_revision: int`, `target_option_id: str | None = None`, `tolerances: list[ToleranceEntry] | None = None`. All fields with camelCase `alias`.
- `BundleResult`: `schema_version`, `applied: bool`, `new_revision: int | None`, `option_id: str | None`, `violations: list[dict]`, `checkpoint_snapshot_id: str | None`.
- `AssumptionEntry` (CMD-V3-02 shape): `key: str`, `value: str | int | float | bool`, `confidence: float` (ge=0, le=1), `source: str`, `contestable: bool = True`, `evidence: str | None = None`.
- All models: `model_config = ConfigDict(populate_by_name=True)`.

### 2. Dry-run / commit / tolerate modes — `app/bim_ai/cmd/apply_bundle.py`

Public signature: `def apply_bundle(doc: Document, bundle: CommandBundle, mode: Literal["dry_run", "commit"]) -> BundleResult`.

Ordered steps:
1. `_validate_assumptions(bundle.assumptions)` — return `assumption_log_required` / `assumption_log_malformed` / `assumption_log_duplicate_key` on failure.
2. Revision guard: `bundle.parent_revision != doc.revision` → `revision_conflict`.
3. `targetOptionId` guard (task 4).
4. Call `engine.try_commit_bundle(doc, bundle.commands)`.
5. Compute `checkpoint_snapshot_id` = SHA-256 of `sort_keys=True` JSON of post-bundle elements wire (use `doc` elements on dry-run failure, `new_doc` elements on success).
6. Mode branch: `dry_run` → return without persisting; `commit` + `ok=True` → return `BundleResult(applied=True, ...)`; `commit` + `ok=False` → return `applied=False`.

`apply_bundle` is **pure** — no DB I/O. Route layer handles persistence.

### 3. `parent_revision` optimistic-lock guard

Guard in step 2 above. Route layer (task 6) must additionally use a conditional DB update (`UPDATE model_records SET revision=N+1 WHERE id=X AND revision=N`); if 0 rows updated, return 409 `revision_conflict`.

### 4. `targetOptionId` routing

- Absent / `None` → proceed normally (commits to current model state; full OPT-V3-01 branching deferred to KRN-V3-04).
- `"main"` → reject with `direct_main_commit_forbidden`.
- Any other non-null value → reject with `option_routing_not_yet_implemented` + pointer to OPT-V3-01.
- Mark both branches `# TODO(OPT-V3-01)`.

### 5. `assumptions` structural validation

`_validate_assumptions` returns advisory dicts. Rules: non-empty array, each entry has `key` (non-empty), `confidence` in [0,1] (else `assumption_log_malformed` with entry index), unique keys (else `assumption_log_duplicate_key`). Kernel validates structure only — semantics live in the agent.

## Implementation rules (§A and §L)

- **No AI in any binary.** `apply_bundle` and `try_commit_bundle` are fully deterministic. No LLM, no inference.
- **API-V3-01 contract.** Update the existing `apply-bundle` stub in `registry.py` — do not register a second descriptor.
- **Assumptions-first (§L).** Kernel rejects structurally invalid bundles. Human scripts and agents satisfy the same contract.
- **Deterministic outputs.** Same bundle + same parent revision → same `BundleResult` every time.
- **No silent main mutation.** `targetOptionId: "main"` always rejected (guard is permanent, not a TODO).
- **Backwards compatibility.** Legacy `POST /api/models/{model_id}/commands/bundle` endpoint must continue to work.
- **Do not split `engine.py`.** CQ-04 is a separate WP. Import `try_commit_bundle` from `bim_ai.engine` as-is; all new logic lives in `app/bim_ai/cmd/`.
