## Full verbatim CQ-01 entry (from `spec/code-quality-tracker.md`)

> **CQ-01 — WebSocket robustness (reconnect + replay buffer + sequence)**
>
> **Status:** `open`
> **Severity:** Medium
> **Blast radius:** Server hub + client WS layer + message envelope. Touches state hydration on reconnect.
>
> **Problem.** `app/bim_ai/hub.py` (~88 LOC) does fire-and-forget broadcast with silent error suppression; a slow consumer is dropped without notice. Client side has no auto-reconnection — a dropped socket leaves the app in `wsOn: false` and requires manual refresh. There is no message sequence number, no replay buffer, no resume protocol. Any drop = state divergence until the user reloads.
>
> **Acceptance criteria.**
>
> - Each broadcast envelope carries a monotonic `seq: number` per `model_id`.
> - Server retains a bounded ring buffer (suggest N=512 or last 30s, whichever larger) of recent deltas keyed by `model_id`.
> - Client persists `lastSeq` per model and on reconnect sends `?resumeFrom=<seq>` (or initial frame). Server replays `seq+1..` if in window, else returns `RESYNC` and client refetches snapshot.
> - Client reconnects with exponential backoff (250ms → 8s, jittered) up to N attempts before surfacing offline state.
> - Server applies per-socket send backpressure: if `send` queue depth exceeds threshold, the socket is closed with code 1011 and the client treats it as a normal reconnect.
> - Tests: integration test in `app/tests/` for replay-window hit, replay-window miss (forces RESYNC), and one for backpressure-induced disconnect.
>
> **Suggested files.**
>
> - `app/bim_ai/hub.py` — add `Hub.publish(model_id, delta) -> seq`, ring buffer, `Hub.resume(model_id, from_seq)`.
> - `app/bim_ai/main.py` (or wherever `/ws/{model_id}` is registered) — read `?resumeFrom=` query, dispatch to `Hub.resume`.
> - `packages/web/src/state/ws.ts` (or equivalent) — backoff, resume protocol, `lastSeq` persistence in store.

## Implementation tasks

### Task 1 — Hub: sequence counter + ring buffer

File: `app/bim_ai/hub.py`

1. Add a per-`model_id` monotonic sequence counter (`_seq: dict[str, int]`). Each call to `broadcast_json` for a `delta` or sequenceable message increments the counter and stamps the payload with `"seq": n`.

2. Add a bounded ring buffer (`_buffer: dict[str, deque[tuple[int, dict]]]`). After stamping a message, append `(seq, payload_copy)` to the deque for `model_id`. Cap at `max(512, entries from the last 30 seconds)`. Use `collections.deque(maxlen=512)` as the base; prune time-based entries on each append (store a `_buffer_ts: dict[str, deque[tuple[float, int]]]` for timestamps if needed, or attach `"ts"` to buffered payloads).

3. Add `Hub.publish(model_id: str, payload: dict) -> int` — the single public entry point for sequenceable messages. It stamps `seq`, appends to buffer, then calls `broadcast_json` internally. Callers in `routes_api.py` that broadcast deltas must be migrated to `hub.publish(...)`.

4. Add `Hub.resume(model_id: str, from_seq: int) -> list[dict] | None`:
   - Returns the list of buffered payloads with `seq > from_seq` if all are still in the ring buffer.
   - Returns `None` if `from_seq` is older than the oldest buffered entry (signals the caller to send `RESYNC`).

5. Add per-socket send backpressure. Track a `_send_queue_depth: dict[int, int]` (keyed on `id(websocket)`). Before each `ws.send_json`, increment depth; decrement in a `finally` block (or after the await). If depth exceeds `BACKPRESSURE_THRESHOLD = 8`, close the socket with code `1011` and call `self.unregister(ws)` instead of sending. Do not suppress this close — the client must see a disconnect and reconnect.

### Task 2 — WebSocket route: resumeFrom handshake

File: `app/bim_ai/routes_api.py` (function `websocket_loop`)

1. Accept a `resumeFrom: int | None = Query(default=None)` parameter on the WS endpoint.

2. After `hub.subscribe(sid, websocket)` and before entering the receive loop:
   - If `resumeFrom` is provided and `resumeFrom >= 0`:
     - Call `hub.resume(sid, resumeFrom)`.
     - If the result is a list (window hit): send each replayed payload in order, then send a `{"type": "replay_done", "modelId": sid, "resumedFrom": resumeFrom}` frame.
     - If the result is `None` (window miss): send `{"type": "RESYNC", "modelId": sid}`. The client must then fetch a fresh snapshot via REST and resume from the new `seq` it receives in the next broadcast.
   - If `resumeFrom` is absent: send the full `snapshot` frame as before (initial connect path).

3. Remove the existing unconditional `snapshot` send — gate it behind `resumeFrom is None`.

### Task 3 — WS client: backoff reconnect + lastSeq

File: `packages/web/src/workspace/useWorkspaceSnapshot.ts`

The existing client creates a raw `WebSocket` in `insertSeedHouse` with no reconnect logic. Replace the inline construction with an extracted `connectWs(modelId, lastSeq)` helper inside the hook.

1. Extract `connectWs(modelId: string, lastSeq: number | null): WebSocket`:
   - Builds URL: `${p}://${host}/ws/${encodeURIComponent(modelId)}${lastSeq !== null ? `?resumeFrom=${lastSeq}` : ''}`.
   - Sets `onopen`, `onmessage`, `onclose`, `onerror`.
   - Returns the `WebSocket` instance.

2. Track `lastSeqRef = useRef<number | null>(null)`. On each incoming message that carries a `seq: number`, update `lastSeqRef.current = seq`.

3. Handle `RESYNC`: on receiving `{type: 'RESYNC'}`, fetch a fresh snapshot via the existing REST path (`/api/models/:id/snapshot?expandLinks=true`), call `hydrateFromSnapshot`, then reset `lastSeqRef.current = null` (the server will stamp the next delta from the current counter).

4. Handle `replay_done`: no-op beyond logging — state is already applied via the replayed delta messages.

5. On `onclose`: implement exponential backoff reconnect. Suggested schedule:
   ```
   attempts:  1    2    3    4    5    6+
   delay ms: 250  500 1000 2000 4000 8000 (capped)
   jitter:   ±20% of delay
   max attempts: 10 before surfacing offline state
   ```
   After max attempts, set `wsOn` to `'offline'` (or keep `false` — the existing `StatusBar` reads `wsOn: boolean`; if you want a richer state surface it in the store; if not, keep `false` and add a console warning).

6. On `onopen`: reset backoff counter to 0, set `wsOn(true)`.

7. The existing `wsRef.current?.close()` cleanup in the `useEffect` teardown remains — call it before reassigning on reconnect.

8. Keep the `VITE_E2E_DISABLE_WS` guard — reconnect must also be suppressed when the flag is set.

### Task 4 — Migrate existing broadcast callsites to `hub.publish`

Search `routes_api.py` (and any other route files that call `hub.broadcast_json`) for delta broadcasts. Any call that emits a `delta` or content-bearing message should be changed to `await hub.publish(model_id, payload)` so it gets stamped with a seq and buffered. Pure presence messages (`presence_state`) do not need seq stamping and can remain as `broadcast_json` calls.

### Task 5 — Tests

File: `app/tests/test_ws_robustness.py` (new file)

Write three pytest-asyncio integration tests using FastAPI's `TestClient` (or `httpx.AsyncClient` with the ASGI app):

1. **`test_replay_window_hit`**: Connect client A, receive snapshot. Server publishes 5 deltas (call `hub.publish` directly). Disconnect client A. Reconnect with `?resumeFrom=<seq_of_last_seen>`. Assert client receives exactly the missed deltas in order, then `replay_done`.

2. **`test_replay_window_miss_forces_resync`**: Same setup. Manually clear the hub's ring buffer (or overflow it) so `from_seq` is before the oldest entry. Reconnect with a stale `resumeFrom`. Assert client receives `{"type": "RESYNC"}`.

3. **`test_backpressure_disconnect`**: Patch `BACKPRESSURE_THRESHOLD = 1`. Connect a client. Have hub attempt to send 2 messages. Assert the socket is closed with code 1011 and the client is unregistered from the hub room.

## Implementation rules

1. **Do not break existing tests.** Run `make verify` before opening a PR. All 106+ existing pytests must pass. The WS integration tests you add are the only new tests.

2. **No seq on presence frames.** `presence_state` and `presence` messages must not be stamped with `seq` or buffered — they are ephemeral and would bloat the ring buffer. Only `delta` and engine-generated content frames go through `hub.publish`.

3. **Ring buffer is in-process only.** Do not introduce Redis, SQLite, or any persistent store for the ring buffer. v3.0 is single-process; persistence is post-v3. The buffer lives on the `Hub` instance.

4. **Keep the Hub stateless across model rooms.** Each `model_id` gets its own seq counter and deque. No shared global counter.

5. **Client backoff must respect `visibilitychange`.** If the browser tab is hidden (`document.visibilityState === 'hidden'`) when a reconnect fires, defer the reconnect until the tab becomes visible again. This prevents silent background reconnects from burning all retry attempts.

6. **Type hygiene.** The new `ws.ts` (or additions to `useWorkspaceSnapshot.ts`) must be TypeScript-strict with no `any` escapes beyond the existing payload parsing. `lastSeq` must be typed as `number | null`, not `number | undefined`.

7. **Prettier and pre-commit hooks run automatically** (per `CLAUDE.md`) — do not run them manually. Do not commit formatting noise.

8. **Keep the diff minimal.** This WP touches three files at its core (`hub.py`, `routes_api.py`, `useWorkspaceSnapshot.ts`). New files are: `app/tests/test_ws_robustness.py`. If you find it cleaner to extract the client reconnect logic into `packages/web/src/state/ws.ts`, do so, but do not reorganise other files — CQ-03 and CQ-05 own that work.
