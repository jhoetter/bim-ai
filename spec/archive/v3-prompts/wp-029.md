def planDoorMesh(door: DoorElem, detail_level: str) -> list[LineSegment]:
    if detail_level == "coarse":
        # return only the door opening outline (two vertical lines at jambs)

    elif detail_level == "medium":

### 3. Engine — `planWindowMesh` detail routing (`engine.py`)

def planWindowMesh(window: WindowElem, detail_level: str) -> list[LineSegment]:

### 6. Tests (`app/tests/test_detail_level_rendering.py`)

Write a new test file. Cover:

- `planDoorMesh(door, 'coarse')` returns fewer line segments than `planDoorMesh(door, 'fine')`.

- Unknown `detail_level` string falls back to `'fine'` behaviour without raising.

## Implementation rules

- **No schema changes.** This WP adds zero new Pydantic fields or TypeScript types. It is purely engine wiring.
- **Coarse < Medium < Fine** in visual complexity. Each level must visibly differ from the previous — write the tests to assert line-segment count differences.
- **Phase filter is orthogonal.** The phase render style (colour + dash pattern) is applied by a separate layer on top of the geometry. Do not conflate phase rendering with detail level rendering.
- **Existing `planWallMesh` is already correct.** Verify it accepts `detailLevel` — only fix if broken. Do not refactor working code.
- **No TSX changes required** for this WP — the `PlanDetailLevelToolbar` (v2-shipped) already dispatches the active view's setter. The fix is engine-only.
