### 2. Route `RoomColorSchemePanel.tsx` to the inspector

Open `packages/web/src/workspace/RoomColorSchemePanel.tsx`. If it currently renders as a standalone panel or floating card:

1. Extract the form fields (the colour scheme controls) into a functional component that can be embedded in an inspector section.
2. In the right-rail inspector (the component shipped by WP-061), add an `InspectorSection` titled "Room Colour Scheme" that renders when the selected element is a room (`selectedElement.kind === 'room'`).
3. Remove the modal/floating wrapper from `RoomColorSchemePanel.tsx`. The panel itself becomes a bare form component.

If `RoomColorSchemePanel.tsx` is already integrated into the inspector, note this in the final report and mark as done.

### 3. Route `ViewTemplateEditPanel.tsx` to the inspector

Same pattern as above. Open `packages/web/src/workspace/ViewTemplateEditPanel.tsx`. If it opens as a modal or separate panel:

1. The view template edit controls should appear in the inspector when the active surface is a view (not a 3D element).
2. Add an `InspectorSection` titled "View Template" that renders when `activeViewType` is one of the 2D view types.
3. Remove the modal/floating wrapper.

If already integrated, note it.

### 4. Audit and route any other modals found in step 1

For each remaining modal found in the audit:
- If it's for a property editor (element properties, view properties, material assignment): route to inspector section.
- If it's a destructive-action confirmation (delete, reset): these are legitimately modal — leave them but ensure they use the correct `<dialog>` element with `aria-modal`.
- If it's a create/save flow (name a new view, save a milestone): these can stay modal for now — note them in the final report as optional follow-ups.

### 5. Confirm the inspector section API

Read the right-rail inspector component carefully to understand the section API. It likely has a pattern like:

```tsx
<InspectorSection title="Room Colour Scheme" defaultExpanded={true}>
  <RoomColourSchemeForm roomId={selectedElement.id} />
</InspectorSection>
```

If no `InspectorSection` component exists, create a minimal one:

```tsx
function InspectorSection({
  title,
  children,
  defaultExpanded = true,
}: {
  title: string;
  children: ReactNode;
  defaultExpanded?: boolean;
}) {
  const [expanded, setExpanded] = useState(defaultExpanded);
  return (
    <div>
      <button
        onClick={() => setExpanded((e) => !e)}
        style={{ /* ... use token-only styles */ }}
      >
        {title}
      </button>
      {expanded && <div>{children}</div>}
    </div>
  );
}
```

## Implementation rules

- **Do not create a new modal under any circumstances in this WP.** If you encounter a case where a modal seems unavoidable, add a note in the final report and defer it — do not add a new modal.
- **Destructive confirmations are exempt.** Delete/reset confirmations that require explicit user acknowledgment may stay as browser `confirm()` or a small `<dialog>` — this WP is about property editors, not confirmation gates.
- **No scope creep.** Do not redesign the inspector, do not add new property sections beyond what's in the audit. If a panel is already in the inspector, leave it.
- **No hex literals.** Any new styled elements use CSS custom properties only.
