## Verbatim tracker entry (§6 T10 VIS-V3-05)

**Status.** `open`.

**Scope.** CAN-V3-02 shipped the `HatchRenderer`, `HatchPatternDef`, and `HatchPicker`. The problem: `hatchPatternId` is `null` on newly created elements because no default is assigned at creation time. This WP adds a category-based fallback map in `HatchRenderer.ts` so walls, floors, and roofs show their standard hatches by default without any explicit material assignment.

**Acceptance.** Place a wall in plan view at 1:50 → brick hatch renders without explicit material assignment. Place a floor slab → concrete hatch renders. Unit test: `resolveHatchPatternId(null, 'wall')` returns `'brick'`.

## Implementation tasks

### 1. Understand the current hatch resolution

Open `packages/web/src/plan/HatchRenderer.ts`. Find the function (or inline logic) that resolves which hatch pattern to render for a given element. It currently does something like:

```typescript
materialsById[materialKey]?.hatchPatternId ?? null
```

When `hatchPatternId` is `null` (no explicit assignment), no hatch renders — the plan canvas shows a flat colour fill. This WP adds a category-based fallback so the standard patterns are always present.

### 2. Add the category default hatch map

In `HatchRenderer.ts`, add a `CATEGORY_DEFAULT_HATCH` constant near the top of the file:

```typescript
const CATEGORY_DEFAULT_HATCH: Partial<Record<string, string>> = {
  wall: 'brick',
  floor: 'concrete',
  roof: 'tile',
  stair: 'concrete',
  slab_edge: 'concrete',
  // railing, door, window intentionally excluded — small profiles don't need hatch
};
```

The pattern IDs (`'brick'`, `'concrete'`, `'tile'`) must match the IDs used by `HatchPatternDef` in `CAN-V3-02`. Read `HatchRenderer.ts` and `draftingStandards.ts` to confirm the exact IDs.

### 3. Update `resolveHatchPatternId` (or equivalent)

Find the function in `HatchRenderer.ts` that resolves the hatch pattern for a given element. Update it to accept an optional `category` argument and apply the fallback:

```typescript
export function resolveHatchPatternId(
  hatchPatternId: string | null | undefined,
  category?: string,
): string | null {
  if (hatchPatternId) return hatchPatternId;
  if (category && CATEGORY_DEFAULT_HATCH[category]) {
    return CATEGORY_DEFAULT_HATCH[category]!;
  }
  return null;
}
```

If the function doesn't exist yet (logic is inline), extract it as a named export — this makes it testable.

### 4. Pass element category to the hatch resolution call site

Find where `resolveHatchPatternId` (or the inline equivalent) is called in `HatchRenderer.ts`. The call site needs access to the element's category. Element category is typically available from the element object (e.g. `element.kind === 'wall'` → category `'wall'`). Pass the category through.

If the call site only has a `materialKey` (not the element itself), trace back where elements are iterated and ensure the category string is threaded through.

### 5. Tests

In the existing test file for `HatchRenderer` (or create `packages/web/src/plan/__tests__/HatchRenderer.test.ts` if none exists), add:

```typescript
describe('resolveHatchPatternId', () => {
  it('returns explicit patternId when set', () => {
    expect(resolveHatchPatternId('stone', 'wall')).toBe('stone');
  });
  it('returns category default for wall when no explicit id', () => {
    expect(resolveHatchPatternId(null, 'wall')).toBe('brick');
  });
  it('returns category default for floor', () => {
    expect(resolveHatchPatternId(null, 'floor')).toBe('concrete');
  });
  it('returns null for categories without a default (e.g. window)', () => {
    expect(resolveHatchPatternId(null, 'window')).toBeNull();
  });
  it('returns null when no category and no explicit id', () => {
    expect(resolveHatchPatternId(null, undefined)).toBeNull();
  });
});
```
