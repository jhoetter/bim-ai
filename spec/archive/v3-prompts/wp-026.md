## Verbatim MRK-V3-01 tracker entry (§6 T3, line 1304)

**MRK-V3-01 — 3D-anchored comments with threading + @mention**

**Status.** `now`.
**Scope.** A `Comment` is entity-anchored or geometry-anchored. Three anchor modes: **(a)** element-anchored; **(b)** point-anchored (3D world coordinate); **(c)** region-anchored (3D bounding volume). Comments are threadable, support `@mention` with email + in-app notification, and re-pin correctly when the model element moves. Surfaces in plan view + 3D view + sheet view. Resolvable.

**Data model.**
```ts
type Comment = {
  id: string;
  modelId: string;
  threadId: string;
  authorId: string;
  body: string;
  mentions: Array<{ userId: string; offsetMm: number }>;
  anchor:
    | { kind: 'element'; elementId: string; offsetLocalMm?: { xMm; yMm; zMm } }
    | { kind: 'point'; worldMm: { xMm; yMm; zMm } }
    | { kind: 'region'; bboxMm: { minMm; maxMm } };
  createdAt: number;
  resolvedAt?: number;
  resolvedBy?: string;
};
```

**Engine.** New `comments` Y.Doc subspace inside the collab session. Element-anchored comments survive geometry edits — when the host element moves, the comment's projected pin moves with it; when deleted, comment becomes "orphaned" with a soft warning chip.

**Effort.** L — 2.5 weeks.

## Implementation tasks

### 1. Data model — Python

Add `CommentAnchor` discriminated union and `Comment` Pydantic model. Place in a new file `app/bim_ai/comments.py`:

```python
from __future__ import annotations
from typing import Literal
from pydantic import BaseModel, ConfigDict, Field

class Vec3Mm(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    x_mm: float = Field(alias="xMm")
    y_mm: float = Field(alias="yMm")
    z_mm: float = Field(alias="zMm")

class ElementAnchor(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    kind: Literal["element"] = "element"
    element_id: str = Field(alias="elementId")
    offset_local_mm: Vec3Mm | None = Field(default=None, alias="offsetLocalMm")

class PointAnchor(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    kind: Literal["point"] = "point"
    world_mm: Vec3Mm = Field(alias="worldMm")

class RegionAnchor(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    kind: Literal["region"] = "region"
    min_mm: Vec3Mm = Field(alias="minMm")
    max_mm: Vec3Mm = Field(alias="maxMm")

CommentAnchor = Annotated[
    ElementAnchor | PointAnchor | RegionAnchor,
    Field(discriminator="kind")
]

class Comment(BaseModel):
    model_config = ConfigDict(populate_by_name=True, extra="ignore")
    id: str
    model_id: str = Field(alias="modelId")
    thread_id: str = Field(alias="threadId")
    author_id: str = Field(alias="authorId")
    body: str
    anchor: CommentAnchor
    created_at: int = Field(alias="createdAt")
    resolved_at: int | None = Field(default=None, alias="resolvedAt")
    resolved_by: str | None = Field(default=None, alias="resolvedBy")
    is_orphaned: bool = Field(default=False, alias="isOrphaned")
```

### 4. Orphan tracking helper (`comments.py`)

Add a pure function:

```python
def mark_orphaned_comments(
    comments: list[Comment],
    deleted_element_ids: set[str],
) -> list[Comment]:
    """Return a new list with element-anchored comments orphaned when their host was deleted."""
    result = []
    for c in comments:
        if (
            c.anchor.kind == "element"
            and c.anchor.element_id in deleted_element_ids
            and not c.is_orphaned
        ):
            result.append(c.model_copy(update={"is_orphaned": True}))
        else:
            result.append(c)
    return result
```

Hook this into the engine's element deletion path so comments are orphaned when their host element is removed.

## Implementation rules

- **yjs subspace.** The `comments` subspace inside the collab Y.Doc is the source of truth for real-time; REST is the persistence/load path. Do not duplicate — the REST layer materialises from yjs on reads if yjs is the canonical store, or vice versa. Choose the simpler path: REST is canonical for v3, yjs is a future enhancement.
- **Thread vs top-level.** Top-level comments have `threadId == id`. Replies have `threadId == parent.id`. Validate this invariant on create.
- **No hex literals.** CSS token variables only in any new TSX.
- **No new element kinds.** `Comment` is not a kernel element — it lives in a separate collection, not in `doc.elements`.
