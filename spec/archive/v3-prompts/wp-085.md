## Verbatim tracker entry (§6 T10 VIS-V3-08)

**Status.** `open`.

**Scope.** When the active view is a plan, section, or elevation, switch the canvas background to warm off-white (`--color-canvas-paper`) — paper on a dark-chrome desk. The 3D perspective viewport keeps its dark background. Add `--color-canvas-paper` token to `tokens-v3.css` and apply it to plan/section/elevation canvas wrappers automatically based on view type.

**Acceptance.** Active plan view → canvas background is warm off-white. Active 3D view → canvas background is `--color-background`. Toggle between them — background switches within one frame. No hex literals in new code.

## Implementation tasks

### 1. Add `--color-canvas-paper` token

Open `packages/design-tokens/src/tokens-v3.css`. Add the paper canvas token to both the light and dark `:root` blocks:

**In `:root, :root[data-theme='light']` block:**
```css
/* VIS-V3-08 — plan/section/elevation canvas background (paper-white) */
--color-canvas-paper: hsl(40 20% 96%);
```

**In `:root[data-theme='dark'], :root.dark` block:**
```css
/* Paper canvas appears slightly warm off-white even in dark mode */
--color-canvas-paper: hsl(40 16% 92%);
```

This is a Layer-B semantic-shell token (it adapts to light/dark mode) — add it in the correct blocks.

### 2. Locate the plan canvas mount component

Search for the component that wraps the plan view canvas. It will be something like `SketchCanvas`, `PlanCanvas`, `CanvasMount`, or the element with `gridArea: 'canvas'` in the workspace. Look in:
- `packages/web/src/plan/`
- `packages/web/src/`
- `packages/web/src/workspace/`

The plan canvas should have a background style or class that sets its colour. Find where that colour is applied.

### 3. Apply `--color-canvas-paper` when view type is 2D

**Strategy A — `data-view-type` attribute (preferred):**

Wherever the canvas wrapper is rendered, add a `data-view-type` attribute reflecting the active view type:

```tsx
<div
  style={{ gridArea: 'canvas', background: 'var(--color-background)' }}
  data-view-type={activeViewType}  // 'plan' | 'section' | 'elevation' | 'drafting_view' | '3d'
>
  {canvas}
</div>
```

Then in the AppShell or Workspace CSS (or a `<style>` tag):
```css
[data-view-type='plan'],
[data-view-type='section'],
[data-view-type='elevation'],
[data-view-type='drafting_view'] {
  background: var(--color-canvas-paper) !important;
}
```

The `!important` override is needed because the inline `background` style has higher specificity. Alternatively, use a CSS custom property to hold the resolved value and override that property rather than using `!important`.

**Strategy B — conditional inline style:**

```tsx
const is2DView = ['plan', 'section', 'elevation', 'drafting_view'].includes(activeViewType ?? '');
<div style={{
  gridArea: 'canvas',
  background: is2DView ? 'var(--color-canvas-paper)' : 'var(--color-background)',
}}>
```

Use whichever strategy is cleaner given the existing code structure.

### 4. Get `activeViewType` in the canvas wrapper

Locate how the active view type is tracked. It will be something like:
```typescript
const activeView = useModelStore((s) => s.activeView);
const viewType = activeView?.kind; // 'viewpoint' | 'plan' | 'elevation' | 'section' | ...
```

Read `packages/core/src/index.ts` to find the view element types (look for `kind: 'viewpoint'` or similar). The kinds that should use paper background: `'plan_view'`, `'section_view'`, `'elevation_view'`, `'drafting_view'`, `'callout_view'`. The 3D perspective view (`'viewpoint'` or `'3d_view'`) keeps the dark background.

### 5. Plan renderer background

If the plan canvas has its own SVG or canvas element with a background fill hardcoded (check for `fill="black"` or similar), update it to use `var(--color-canvas-paper)` as the fill. The plan SVG root `<rect>` background, if present, should match.

Also check `packages/web/src/plan/` for any CSS that sets a background directly on the plan canvas — update those to `var(--color-canvas-paper)`.

### 6. Tests

In `packages/web/src/workspace/AppShell.test.tsx` or a new test file, add tests:
- When `activeViewType='plan'`, canvas wrapper has `background: var(--color-canvas-paper)` or `data-view-type='plan'`
- When `activeViewType='3d'`, canvas wrapper has `background: var(--color-background)` or no paper mode attribute

```typescript
it('sets paper background for plan view', () => {
  render(<Workspace activeViewType="plan" ... />);
  const canvas = screen.getByTestId('canvas-area');
  expect(canvas).toHaveAttribute('data-view-type', 'plan');
  // or check computed style
});
```

## Implementation rules

- **No hex literals in component code.** Paper canvas colour comes from `var(--color-canvas-paper)` only — defined once in tokens-v3.css.
- **No scope creep.** Do not add a dark/light mode toggle for the user, do not redesign the toolbar, do not change anything about the 3D renderer's background (Three.js setClearColor). Only the CSS background of the canvas wrapper element changes.
- **Smooth transition (optional but desirable).** If switching between 3D and plan views already has a transition, the background should also transition: `transition: background 120ms var(--ease-paper)`.
