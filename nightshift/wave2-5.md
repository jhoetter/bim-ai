### 1.1 — PLN-02: Crop region UI toggle on plan canvas

**Tracker:** PLN-02. `cropMinMm` / `cropMaxMm` data already exists on `plan_view`; only used in sheet viewports today. Add canvas drag handles.

**Scope:**

1. In `packages/web/src/plan/PlanCanvas.tsx`, when `view.cropMinMm` and `view.cropMaxMm` are set:
   - Render a rectangular crop frame as a bold dashed line on the plan canvas
   - Render 8 drag handles (corners + edges) on the frame
   - Dragging a corner handle resizes the crop (commits `UpdateElementProperty` on `cropMinMm` / `cropMaxMm`)
   - Dragging an edge handle resizes that edge
   - Dragging the frame body moves the whole crop

2. View properties dropdown (find the existing one in `Workspace.tsx` or sidebar): add a checkbox "Crop View" (toggles `view.cropEnabled`) and "Crop Region Visible" (toggles whether the frame is drawn even when crop is off).

3. When crop is enabled, plan rendering clips elements outside the crop (or fades them — your call; clipping is preferred).

4. Tests: `packages/web/src/plan/cropRegionDrag.test.tsx` — drag fires correct command.

**Acceptance.** A plan view with `cropMinMm: {0, 0}, cropMaxMm: {7000, 8000}` renders a dashed crop frame with handles; dragging a corner resizes; toggling "Crop View" off shows everything outside.

**Effort:** 3-4 hours.

### 1.2 — PLN-01: Dimension / tag automation

**Tracker:** PLN-01. Manual placement works; auto-dimension chains and auto-tag-everything modes missing.

**Scope:**

1. New helper module `packages/web/src/plan/autoDimension.ts`:
   - `generateAutoDimensionChain(view, axis: 'x' | 'y'): CreateDimensionCmd[]` — finds all walls on the active level perpendicular to `axis`, computes their endpoint positions, generates a chain of dimensions between consecutive endpoints
   - `generateAutoTags(view): { CreateRoomTagCmd | CreateDoorTagCmd | … }[]` — finds all rooms / doors / windows in the view's crop, generates tag commands at their centroids

2. New tool buttons in Annotate ribbon:
   - "Auto Dimension X" — fires `generateAutoDimensionChain(view, 'x')`, commits via bundle
   - "Auto Dimension Y" — same for Y
   - "Auto Tag — Rooms" — generates room tags
   - "Auto Tag — Doors" — generates door tags
   - "Auto Tag — Windows" — generates window tags
   - Each button has a small undo-style hint: "Generates N dimensions/tags. Run again to regenerate."

3. Re-running an auto-tool removes existing auto-generated dimensions/tags first (mark them with `autoGenerated: true` and clear those before re-emitting) — prevents duplication.

4. Tests:
   - `packages/web/src/plan/autoDimension.test.ts` — chain generation produces correct dimension commands for a sample model
   - `packages/web/src/plan/autoTags.test.ts` — room/door/window tag generation

**Acceptance.** Click "Auto Dimension X" on the seeded ground-floor plan view → a horizontal dimension chain appears at the south edge with each wall endpoint dimensioned. Click "Auto Tag — Rooms" → every room gets a tag at its centroid. Re-running clears+regenerates.

**Effort:** 5-6 hours.

### 1.3 — ANN-01: Detail components (`detail_line`, `detail_region`, `text_note`)

**Tracker:** ANN-01.

**Scope:**

Three new view-local element kinds. They are pure 2D — not visible in 3D.

1. **`detail_line`** in `core/index.ts`:

   ```ts
   {
     kind: 'detail_line';
     id: string;
     hostViewId: string;
     pointsMm: { xMm: number; yMm: number }[];
     strokeMm?: number;
     colour?: string;
     style?: 'solid' | 'dashed' | 'dotted';
   }
   ```

2. **`detail_region`**:

   ```ts
   {
     kind: 'detail_region';
     id: string;
     hostViewId: string;
     boundaryMm: { xMm: number; yMm: number }[];
     fillColour?: string;
     fillPattern?: 'solid' | 'hatch_45' | 'hatch_90' | 'crosshatch' | 'dots';
     strokeMm?: number;
     strokeColour?: string;
   }
   ```

3. **`text_note`**:

   ```ts
   {
     kind: 'text_note';
     id: string;
     hostViewId: string;
     positionMm: { xMm: number; yMm: number };
     text: string;
     fontSizeMm: number;
     anchor?: 'tl' | 'tc' | 'tr' | 'cl' | 'c' | 'cr' | 'bl' | 'bc' | 'br';
     rotationDeg?: number;
     colour?: string;
   }
   ```

4. Mirror in `app/bim_ai/elements.py` and `commands.py`. Engine: validation that `hostViewId` resolves to a view (plan / section / elevation).

5. **Plan rendering** in `packages/web/src/plan/planProjection.ts`:
   - `detail_line`: render polyline at the points
   - `detail_region`: render filled polygon with the pattern (use SVG patterns for hatch)
   - `text_note`: SVG `<text>` at the position with the specified font size and anchor

6. **Tools** (Annotate ribbon): "Detail Line", "Detail Region (Sketch)", "Text Note". Detail-region tool eventually uses SKT-01 (when the agent's WP ships) but for the load-bearing slice, accept a single freehand or click-polygon authoring mode.

7. Tests: vitest for each rendering path.

**Acceptance.** Adding a `text_note` to a plan view renders the text at the correct position and anchor. `detail_line` renders as a polyline; `detail_region` renders as a filled polygon with the chosen pattern. None are visible in 3D views.

**Effort:** 5-6 hours.
